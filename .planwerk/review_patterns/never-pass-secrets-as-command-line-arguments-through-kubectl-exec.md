# Review Pattern: Never pass secrets as command-line arguments through kubectl exec

**Review-Area**: security
**Detection-Hint**: Search for patterns where shell variables containing secrets are interpolated into `kubectl exec` argument lists (e.g., `kubectl exec ... -- cmd "password=${secret_var}"`). The expanded value will be visible in `/proc/<pid>/cmdline` on both the host and inside the pod.
**Severity**: WARNING
**Occurrences**: 3

## What to check

Any `kubectl exec` call where a secret value appears as a positional argument or environment variable passed via `env`. Secrets should be generated inside the pod (e.g., `sh -c 'bao kv put path key=$(openssl rand -base64 32)'`) or passed via stdin/tmpfs, never as CLI arguments.

## Why it matters

Command-line arguments are visible to any process that can read `/proc/<pid>/cmdline` on the host or in the pod's PID namespace. This creates a transient but real secret exposure window, contradicting any claim that passwords are not echoed.

## Examples from external reviews

### CC-0009 — greptile-apps[bot]
- **Feedback**: Passwords generated by `generate_password` are expanded in the calling shell and then forwarded to `bao kv put` as positional arguments via `kubectl exec`. On Linux this means the cleartext value appears in `/proc/<pid>/cmdline`.
- **What was missed**: Any `kubectl exec` call where a secret value appears as a positional argument or environment variable passed via `env`. Secrets should be generated inside the pod (e.g., `sh -c 'bao kv put path key=$(openssl rand -base64 32)'`) or passed via stdin/tmpfs, never as CLI arguments.
- **Fix**: Refactored to generate passwords inside the pod via `sh -c` and `openssl rand`, so secret values never appear as command-line arguments to kubectl exec.

### CC-0009 — greptile-apps[bot]
- **Feedback**: Each unseal key is extracted from the Kubernetes Secret and passed directly as a positional argument to `bao operator unseal`. During the `kubectl exec` call, the cleartext base64-encoded key appears in `/proc/<pid>/cmdline` on the operator workstation and in the container's process table on the node.
- **What was missed**: Any shell command where sensitive values (unseal keys, tokens, passwords, init output) are passed as command-line arguments rather than piped via stdin or read from a file. Includes both direct invocations and kubectl exec wrappers.
- **Fix**: Changed to pipe keys via stdin: `echo "${key}" | kube_exec_stdin ... bao operator unseal -` and replaced `kubectl create secret --from-literal` with `kubectl apply -f -` using a piped manifest.

### CC-0009 — greptile-apps[bot]
- **Feedback**: `bao_exec` and `bao_exec_stdin` inject the vault root token using the `env` wrapper as a positional argument to `kubectl exec`. The expanded token value is visible in `/proc/<pid>/cmdline` on the operator workstation and in the container's process table.
- **What was missed**: Check whether sensitive values (root tokens, passwords, API keys) are passed as positional or flag arguments to shell commands, `kubectl exec`, or wrapper functions. These are visible in process listings to any user on the host. Prefer environment variables injected via files, stdin, or mounted secrets.
- **Fix**: Added TODO comments documenting the `/proc/<pid>/cmdline` exposure and the intended future fix to read the token from a mounted file or K8s Secret inside the pod.
